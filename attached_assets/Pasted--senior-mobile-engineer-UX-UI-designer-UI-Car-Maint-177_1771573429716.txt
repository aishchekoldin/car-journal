Ты — senior mobile engineer + UX/UI designer. Собери полноценный UI мобильного приложения “Car Maintenance Journal” для ведения журнала обслуживания автомобиля. Фокус: мобильный интерфейс и навигация. Бэкенд и реальная БД не обязательны — достаточно мок-данных и локального хранения (placeholder).

1) Платформа и стек
- React Native + TypeScript (Expo) — предпочтительно, чтобы приложение сразу запускалось в Replit.
- Навигация: React Navigation (Bottom Tabs + Stack внутри вкладок).
- UI: аккуратный минималистичный стиль (карточки, отступы, типографика), можно использовать React Native Paper или NativeBase, но без визуальной перегрузки.
- Данные: mock state + опционально AsyncStorage (как временная замена БД). Пометь кодом и комментариями “placeholder”.

2) Обязательное требование — нижний navbar (Bottom Tab Bar)
Внизу экрана всегда доступен таб-бар с 4 пунктами:
A) Главная
B) Журнал работ
C) Статистика
D) Профиль

Каждый пункт — отдельный экран (Stack внутри таба допустим).

3) Экран “Главная” (Dashboard)
Цель: быстрый обзор машины и последней траты.
UI:
- Hero-блок с фото автомобиля (по умолчанию “Skoda Yeti”), поддержи замену фото позже.
- Карточка “Информация о машине”: марка/модель, год (опционально), текущий пробег (последний введённый).
- Карточка “Последняя трата”:
  - дата
  - тип события (плановое/внеплановое) — badge
  - краткое название записи
  - итоговая сумма
  - CTA “Открыть запись” (переход в детали записи в “Журнал работ”)
- Мини-карточка KPI (по желанию): “Среднее в месяц” и/или “В год” — показывай “нет данных”, если записей нет.

4) Экран “Журнал работ” (лог записей + создание)
Цель: вести список событий обслуживания и создавать новые записи.
Структура данных записи:
- id
- date (по умолчанию сегодня)
- mileageKm (обязательное)
- eventType: planned | unplanned (обязательное)
- title (обязательное) — например “ТО 120 000”
- items: массив подпунктов (обязательное условие!)
  - itemId
  - name (обязательное) — “работа”, “покупка масла”, “лампа”, “мойка” и т.п.
  - cost (обязательное, >= 0)
- totalCost — вычисляется как сумма items
- currency — по умолчанию ₽ (но как параметр профиля)

UI “Журнал работ”:
- Вверху экран: кнопка “Добавить запись” (FAB или кнопка).
- Список записей (по умолчанию новые сверху):
  - дата, пробег, badge типа, title, totalCost
  - нажатием открывается экран “Детали записи”
- Поиск по title/подпунктам (UI + простая фильтрация в состоянии).
- Фильтр по типу (все/плановое/внеплановое).
- Пустое состояние: заглушка + “Добавить первую запись”.

Создание записи (экран или модалка + отдельный экран для удобства):
- Поля:
  - Пробег (число, км) — обязательное, > 0
  - Тип события planned/unplanned — обязательное
  - Название записи — обязательное
  - Дата — date picker, дефолт сегодня
- Блок “Подпункты” (items):
  - Список добавленных подпунктов
  - Кнопка “Добавить подпункт”
  - Каждый подпункт: название + стоимость
  - Возможность удалить подпункт
  - Итог “Сумма по записи” (live)
- Валидация:
  - обязательные поля
  - хотя бы 1 подпункт
  - cost >= 0, mileage > 0
- По “Сохранить” запись добавляется в mock/AsyncStorage, возвращаемся в список, история обновляется.

Экран “Детали записи”:
- Шапка: дата, пробег, тип, title, total
- Ниже: список подпунктов (items) с ценами
- (Опционально) кнопки “Редактировать” и “Удалить” — можно пока как заглушки.

5) Экран “Статистика”
Цель: базовые показатели расходов.
UI-компоненты:
- График “Траты по месяцам” (последние 6–12 месяцев):
  - можно простой bar/line chart через react-native-svg-charts или victory-native
  - если не хочешь тянуть библиотеку — сделай “плейсхолдер графика” + подготовь структуру данных
- KPI карточки:
  - “Средние затраты в месяц”
  - “Средние затраты в год”
- Дополнительно (опционально):
  - “Плановое vs внеплановое” (пирог/бейджи) как UI-заглушка
Состояния:
- Если данных нет — пустая статистика с подсказкой добавить записи.

6) Экран “Профиль”
Цель: управление машиной и аккаунтом.
UI-секции:
A) Профиль пользователя:
- Имя
- Почта
- Операции с аккаунтом (UI-кнопки, даже если заглушки):
  - “Выйти”
  - “Удалить аккаунт”
  - “Сменить пароль” / “Сменить почту” (можно как disabled/placeholder)
B) Машина:
- “Добавить/изменить машину”
- Поля:
  - Марка/модель (например, Skoda Yeti)
  - VIN номер
  - Год выпуска (опционально)
  - Фото машины (выбор из галереи/камера — можно placeholder UI)
  - Валюта (₽/€/$) — влияет на отображение стоимостей
- Кнопка “Сохранить”
Сохраняй данные в AsyncStorage как placeholder.

7) Визуальный стиль и UX
- Mobile-first: удобные тапы, крупные кнопки, читаемые поля, нормальные состояния ошибок.
- Используй consistent spacing, карточки, бейджи статусов.
- Поддержи тёмную тему (опционально) или хотя бы готовность к ней (переменные/тема).

8) Архитектура кода
- Раздели на папки: screens/, components/, navigation/, types/, storage/.
- Типы TS обязательно (Record, RecordItem, CarProfile, UserProfile).
- Storage слой (AsyncStorage) через небольшие функции:
  - loadRecords(), saveRecords()
  - loadProfile(), saveProfile()
- KPI и статистику считай на фронте (чистые функции): totals by month, avg per month, avg per year.

Результат:
- Expo RN проект, который запускается сразу.
- Bottom Tab навигация с 4 экранами.
- UI главной, журнала (включая multi-item записи), статистики (график/плейсхолдер), профиля (данные пользователя + машина/VIN + валюты + операции аккаунта).
- Минимальная, но рабочая логика на фронте: создание записи, сохранение в AsyncStorage, обновление списков и KPI.
- Без внешнего бэкенда и без авторизации (кнопки аккаунта могут быть заглушками, но UI должен быть).
